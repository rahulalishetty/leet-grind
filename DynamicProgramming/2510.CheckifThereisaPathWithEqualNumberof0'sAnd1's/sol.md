# Counting Good Binary Strings

## Problem Restatement

You are given four integers:

- `minLength`
- `maxLength`
- `oneGroup`
- `zeroGroup`

A binary string is called **good** if:

1. its length lies in the range `[minLength, maxLength]`
2. every maximal block of consecutive `1`s has length that is a multiple of `oneGroup`
3. every maximal block of consecutive `0`s has length that is a multiple of `zeroGroup`

You must return the number of good binary strings, modulo:

```text
10^9 + 7
```

Also note:

> `0` is considered a multiple of every integer

That statement mainly matters conceptually when a string does **not** contain a block of a certain type at all. For example, a string with no `1`s does not violate the `oneGroup` condition.

---

## First Intuition

A valid binary string is made of **alternating runs**:

- a run of `0`s
- then a run of `1`s
- then a run of `0`s
- and so on

Each run length must obey a divisibility rule:

- every `0`-run length must be a multiple of `zeroGroup`
- every `1`-run length must be a multiple of `oneGroup`

So a good string is not about arbitrary bit-by-bit choices.
It is really about building a string out of valid chunks while preserving the run structure.

That is the key simplification.

---

## Why Brute Force Is Not Practical

If we tried all binary strings of lengths from `minLength` to `maxLength`, then for each length `L` we would have:

```text
2^L
```

possible strings.

Even checking a single string is easy, but the number of strings explodes exponentially.

So we need a structural DP rather than enumeration.

---

## Core Insight: Build Strings by Appending Valid Chunks

Instead of thinking in terms of arbitrary complete strings, think about how a valid string can be formed from a shorter valid string.

Suppose you already have a good string.

### If you append `zeroGroup` zeros

Then:

- if the current string already ends with `0`, the trailing `0` block grows by exactly `zeroGroup`
- if it ends with `1` or is empty, a new `0` block of size `zeroGroup` begins

In both cases, the size of every `0` block remains a multiple of `zeroGroup`.

### If you append `oneGroup` ones

Symmetrically:

- if the string ends with `1`, the trailing `1` block grows by exactly `oneGroup`
- if it ends with `0` or is empty, a new `1` block of size `oneGroup` begins

Again, every `1` block remains a multiple of `oneGroup`.

So every good string can be generated by repeatedly appending one of two valid chunks:

- `zeroGroup` copies of `'0'`
- `oneGroup` copies of `'1'`

This is the central DP idea.

---

## Why Appending Only One Chunk at a Time Is Enough

A block of zeros may have length:

```text
zeroGroup, 2 * zeroGroup, 3 * zeroGroup, ...
```

and a block of ones may have length:

```text
oneGroup, 2 * oneGroup, 3 * oneGroup, ...
```

At first, it may seem that the DP must consider all such multiples directly.

But that is unnecessary.

For example, a zero block of length `3 * zeroGroup` can be viewed as:

- append `zeroGroup` zeros
- then append another `zeroGroup` zeros
- then append another `zeroGroup` zeros

As long as the previous string is already good, each append operation preserves validity.

So a large valid block is just repeated application of the same legal extension.

That means the DP only needs the two atomic transitions:

- add `zeroGroup` zeros
- add `oneGroup` ones

This is what makes the solution linear.

---

## DP Definition

Let:

```text
total[i] = number of good binary strings of exact length i
```

We define:

```text
total[0] = 1
```

This corresponds to the empty string.

Why count the empty string here?

Not because it is part of the final answer, but because it serves as the natural construction base:

- from the empty string, appending `zeroGroup` zeros creates the first valid all-zero block
- from the empty string, appending `oneGroup` ones creates the first valid all-one block

So `total[0] = 1` is a standard DP base.

---

## Transition

To build a good string of length `i`:

### Option 1: end by appending a zero chunk

If `i >= zeroGroup`, then any good string of length:

```text
i - zeroGroup
```

can be extended by appending `zeroGroup` zeros.

So this contributes:

```text
total[i - zeroGroup]
```

ways.

### Option 2: end by appending a one chunk

If `i >= oneGroup`, then any good string of length:

```text
i - oneGroup
```

can be extended by appending `oneGroup` ones.

So this contributes:

```text
total[i - oneGroup]
```

ways.

Therefore:

```text
total[i] =
    (i >= zeroGroup ? total[i - zeroGroup] : 0)
  + (i >= oneGroup  ? total[i - oneGroup]  : 0)
```

taken modulo `10^9 + 7`.

---

## Why This Recurrence Is Correct

This recurrence has two parts to justify:

### 1. Every transition it counts is valid

If a string of length `i - zeroGroup` is already good, appending `zeroGroup` zeros preserves the divisibility rule for zero runs.
The same logic applies for appending `oneGroup` ones.

So every counted transition indeed produces a good string.

### 2. Every good string is counted

Take any good string of length `i`.

Look at its last character.

- If it ends with `0`, then its last zero run has length that is a multiple of `zeroGroup`
- therefore you can remove exactly `zeroGroup` zeros from the end
- the remaining prefix is still a good string

Similarly, if it ends with `1`, you can remove exactly `oneGroup` ones from the end, and the remaining prefix is still good.

So every good string of length `i` comes from exactly one of the two transitions above.

That proves the recurrence is exact.

---

## Another View: End-State Interpretation

You can also think of `total[i]` as the total number of valid ways to reach length `i` in a graph where:

- from length `x`, you can go to `x + zeroGroup`
- from length `x`, you can go to `x + oneGroup`

Every path corresponds to a sequence of appended chunks.

This is just another way of seeing the same DP.

## Full Java Code

```java
class Solution {
    private static final int MOD = 1_000_000_007;

    public int goodBinaryStrings(int minLength, int maxLength, int oneGroup, int zeroGroup) {
        long[] total = new long[maxLength + 1];
        total[0] = 1;  // empty string for construction

        long ans = 0;

        for (int i = 1; i <= maxLength; i++) {
            long ways = 0;

            if (i >= zeroGroup) {
                ways += total[i - zeroGroup];
            }
            if (i >= oneGroup) {
                ways += total[i - oneGroup];
            }

            total[i] = ways % MOD;

            if (i >= minLength) {
                ans = (ans + total[i]) % MOD;
            }
        }

        return (int) ans;
    }
}
```

## Detailed Code Walkthrough

## 1. Mod constant

```java
private static final int MOD = 1_000_000_007;
```

The number of good strings can become very large, so all calculations must be done modulo `10^9 + 7`.

---

## 2. DP array

```java
long[] total = new long[maxLength + 1];
```

`total[i]` stores the number of good binary strings of exact length `i`.

We use `long` for intermediate arithmetic safety before taking modulo.

---

## 3. Base case

```java
total[0] = 1;
```

This is the empty string.

It is not necessarily counted in the final answer unless the allowed length interval included `0`, which it does not here.
But it is essential as the construction starting point.

Without this base case, strings like:

- `"000...0"` of length `zeroGroup`
- `"111...1"` of length `oneGroup`

would not be generated.

---

## 4. Iterate over all lengths

```java
for (int i = 1; i <= maxLength; i++) {
    long ways = 0;
```

We compute the number of good strings for each length from `1` up to `maxLength`.

---

## 5. Add the zero-chunk contribution

```java
if (i >= zeroGroup) {
    ways += total[i - zeroGroup];
}
```

If length `i` is large enough, we can append a zero chunk of size `zeroGroup` to any good string of length `i - zeroGroup`.

---

## 6. Add the one-chunk contribution

```java
if (i >= oneGroup) {
    ways += total[i - oneGroup];
}
```

Similarly, append a one chunk of size `oneGroup` to any good string of length `i - oneGroup`.

---

## 7. Store the result modulo MOD

```java
total[i] = ways % MOD;
```

Now `total[i]` holds the number of good strings of exact length `i`.

---

## 8. Add to final answer if length is within range

```java
if (i >= minLength) {
    ans = (ans + total[i]) % MOD;
}
```

The problem asks for good strings whose lengths are between `minLength` and `maxLength`, inclusive.

So once `i` enters that range, we add `total[i]` to the answer.

---

## 9. Final return

```java
return (int) ans;
```

The final count is returned modulo `10^9 + 7`.

## Worked Example 1

Take:

```text
minLength = 2
maxLength = 3
oneGroup = 2
zeroGroup = 1
```

This means:

- every zero run may have length `1, 2, 3, ...`
- every one run must have length `2, 4, 6, ...`

### Initialize

```text
total[0] = 1
```

---

### Length 1

- can append `zeroGroup = 1` zeros to length `0`
- cannot append `oneGroup = 2` ones

So:

```text
total[1] = total[0] = 1
```

Valid string:

```text
0
```

---

### Length 2

- append one zero chunk to any good string of length `1` -> `total[1] = 1`
- append one one chunk to any good string of length `0` -> `total[0] = 1`

So:

```text
total[2] = total[1] + total[0] = 1 + 1 = 2
```

Valid strings:

```text
00
11
```

---

### Length 3

- append one zero chunk to good strings of length `2` -> `2`
- append one one chunk to good strings of length `1` -> `1`

So:

```text
total[3] = total[2] + total[1] = 2 + 1 = 3
```

Valid strings:

```text
000
011
110
```

---

### Final answer

We need lengths in `[2, 3]`:

```text
total[2] + total[3] = 2 + 3 = 5
```

So the answer is:

```text
5
```

## Worked Example 2

Take:

```text
minLength = 3
maxLength = 3
oneGroup = 2
zeroGroup = 2
```

Now every block length must be a multiple of `2`.

Let us compute.

### `total[0] = 1`

### Length 1

Cannot append a zero chunk of size `2`, cannot append a one chunk of size `2`:

```text
total[1] = 0
```

### Length 2

Can append:

- `00` from empty
- `11` from empty

So:

```text
total[2] = total[0] + total[0] = 2
```

### Length 3

Can only try:

- from `total[1]` via zero chunk
- from `total[1]` via one chunk

But `total[1] = 0`, so:

```text
total[3] = 0
```

That is correct: no binary string of length `3` can have all zero-runs and one-runs of even length.

Answer:

```text
0
```

## Why No Extra ŌĆ£Ending BitŌĆØ State Is Needed

At first, it might look like we should track whether a string ends in `0` or `1`.

That would certainly work, but it is not necessary here.

Why?

Because appending exactly `zeroGroup` zeros is always valid for **any** already-good string, regardless of what it ends with:

- if it ends with `0`, the zero run simply grows by another multiple of `zeroGroup`
- if it ends with `1` or is empty, a new zero run of size `zeroGroup` starts

So the same transition can start from any good string.

This is why the 1D recurrence is enough.

That is an important simplification.

---

## Time Complexity

We iterate once from `1` to `maxLength`.

Inside each iteration we do only constant work.

So:

```text
Time Complexity = O(maxLength)
```

---

## Space Complexity

We store the DP array `total` of size `maxLength + 1`.

So:

```text
Space Complexity = O(maxLength)
```

---

## Can Space Be Reduced Further?

Yes, but only if `oneGroup` and `zeroGroup` are small enough that rolling storage is convenient.

Since the transition only needs:

- `total[i - zeroGroup]`
- `total[i - oneGroup]`

you could optimize space using a circular buffer of size roughly:

```text
max(oneGroup, zeroGroup)
```

But in practice, the array-based solution is clearer and already efficient.

So the full array is the best practical choice unless memory constraints are unusually tight.

---

## Final Takeaway

The entire problem becomes simple once you stop thinking about arbitrary binary strings and instead think about **building valid strings by appending valid chunks**.

A good string can be constructed by repeatedly appending:

- `zeroGroup` zeros
- or `oneGroup` ones

to an already-good shorter string.

That leads directly to the DP:

```text
total[i] =
    (i >= zeroGroup ? total[i - zeroGroup] : 0)
  + (i >= oneGroup  ? total[i - oneGroup]  : 0)
```

Then sum `total[i]` over all lengths in `[minLength, maxLength]`.

### Final complexities

- **Time Complexity:** `O(maxLength)`
- **Space Complexity:** `O(maxLength)`

The key insight is that valid run lengths being multiples of the group sizes allows every legal block to be built incrementally from fixed-size chunks.
